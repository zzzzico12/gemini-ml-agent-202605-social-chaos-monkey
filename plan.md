# Social Chaos Monkey 実装計画書

## 1. プロジェクト憲章 (つくる・まわす・とどける)
- **つくる (Create)**: Gemini 2.5 Flash を中核に、単なるルールベースではない「自律的判断」を行うマルチエージェントを構築。フェイクニュースを「システムの脆弱性」と捉え、カオスエンジニアリングとしてシミュレーションする独創的な設計。
- **まわす (Run)**: プロンプトやエージェントのパラメータをコードとして管理。GitHub Actions 等を用いて、現実のトレンドデータを反映したモデルの継続的改善（CI/CP: Continuous Prompting）サイクルを構築。
- **とどける (Deliver)**: Google Cloud (Cloud Run, Firestore, Pub/Sub) を活用。スケーラブルで堅牢なインフラ上に、実務担当者が「対策（パッチ）」を検証できる本番品質のダッシュボードを提供。

## 3. フェーズ別実装ロードマップ

### Phase 1: 【つくる】コア・シミュレーションの高度化
- [x] **伝播ロジックの精密化**: `following`（フォロー関係）に基づいたグラフ構造を考慮したコンテキスト注入の実装。
- **介入（パッチ）機能**: シミュレーションの途中で「公式ファクトチェック」などの介入情報を投入し、拡散がどう抑制されるかをテストする機能。
- **エージェントの必然性強化**: 各個体が過去の感情を記憶し、時間経過と共に態度を硬化/軟化させるステートフルな思考の実装。

### Phase 2: 【とどける】スケールの適正化と本番品質の追求
- **Google Cloud Pub/Sub の統合**: 複数のエージェント（数十程度）を非同期で扱う際、APIタイムアウトを避けるためのイベント駆動型アーキテクチャへの移行。
- **Cloud Run / Jobs へのデプロイ**: インフラのサーバーレス化による、スパイク的な負荷への対応。
- **並列実行の最適化**: セマフォを用いた Gemini API のレートリミット管理。

### Phase 3: 【とどける】実務に耐えうる観測可能性
- **Social Vulnerability Score (SVS) の計算ロジック**: 拡散速度、センチメント、到達範囲から「デマ耐性」を算出するアルゴリズムの実装。
- **シミュレーション履歴 API**: 過去の `session_id` を指定して、拡散の推移を時系列で取得するエンドポイント。
- **ダッシュボード UI (React)**: 拡散のトポロジー（ネットワーク図）とメトリクスをリアルタイム表示する管理画面の開発。

### Phase 4: 【まわす】DevOps と継続的 AI 改善
- **CI/CD & CI/CP**: GitHub Actions を用いた自動デプロイ。プロンプトの変更が拡散シミュレーションの精度にどう影響するかを自動評価する回帰テストの実装。
- **トレンドフィードバック**: Vertex AI Search や External APIs を活用し、現実の炎上事例を元にエージェントのプロンプトを自動更新する「継続的学習」の仕組み。
