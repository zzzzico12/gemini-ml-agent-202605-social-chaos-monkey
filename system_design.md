# システム基本設計書：Social Risk Simulator

## 1. システム概要 & アーキテクチャ
数十のAIエージェント（架空のSNSユーザー）が自律的にタイムラインを構築し、フェイクニュースの拡散パターンをシミュレートするスケーラブルなサーバーレスシステム。

### 【アーキテクチャ概要】

```text
[ フロントエンド (Next.js/React) ]  ←(WebSockets/SSE)─ [ Firestore / Vertex AI ]
         │ (HTTPS)
         ▼
[ API ゲートウェイ (Cloud Run) ]
         │ (非同期イベント発火)
         ▼
[ Cloud Pub/Sub ] ────▼ (並列分散メッセージ)
               [ Agent Runner (Cloud Run / Job) ] ──▶ [ Vertex AI (Gemini Flash) ]
                        │ (タイムライン書き込み)
                        ▼
               [ Firestore (NoSQL) ]

```

## 2. 「つくる」：エージェントの自律駆動設計

各AIエージェントは独立したコンテナプロセス、またはPub/Subのサブスクライバーとして並列駆動します。エージェントの「脳」には、高速かつ自律的な判断に優れた **Gemini 2.5 Flash** を採用します。

### エージェントのデータモデル（Firestore）

```json
{
  "agent_id": "agent_9981",
  "name": "タカシ@情報通",
  "persona": {
    "gullibility": 0.8,    // デマの信じやすさ (0.0 - 1.0)
    "influence": 0.6,      // 影響力/フォロワー数スコア
    "interests": ["テクノロジー", "政治", "ゴシップ"],
    "political_bias": "Neutral"
  },
  "following": ["agent_1234", "agent_5678"]
}

```

### 自律判断アルゴリズム（思考ループ）

1. **イベント検知:** フォローしているアカウントが発言（ポスト）すると、`Cloud Pub/Sub` 経由でエージェントに通知が届く。
2. **コンテキスト解析:** エージェントは自身の「ペルソナ」と「ポストの内容」を Gemini に入力。
3. **意思決定:** Gemini は以下の JSON フォーマットで自律的に次の行動を出力する。

```json
{
  "action": "RETWEET" | "REPLY" | "IGNORE",
  "reply_content": "え、これマジ！？本当ならヤバすぎるだろ...", // REPLYの場合のみ
  "internal_emotion": "Surprised and anxious" // 次の行動への内部状態の遷移
}

```

## 3. 「まわす」：AIを継続的に改善する CI/CD パイプライン

人間の心理や炎上のトレンドは変化するため、シミュレーションの精度を保つための自動改善ループを構築します。

### CI/CD パイプライン構成（GitHub Actions × Artifact Registry）

```text
[コード変更 / トレンドデータ更新] 
         │
         ▼
 1. GitHub Actions 起動
         │
         ▼
 2. 評価テスト (Validation Test) ──▶ 【テスト内容】過去の実例データと
         │                             シミュレーション結果の乖離が
         ▼                             閾値（誤差15%以内）以下か検証
 3. ビルド & 登録 (Docker Image ─▶ Artifact Registry)
         │
         ▼
 4. 自動デプロイ (Cloud Run への無停止デプロイ)

```

### 継続的改善（CD）のポイント

* **プロンプトのコード管理（GitOps）:** エージェントの行動を定義するシステムプロンプト（System Instructions）をリポジトリ内で管理。
* **自動バックテスト:** 新しいプロンプトをデプロイする前に、過去に実際に起きた有名な炎上ニュースの拡散グラフを再現できるか自動テストを実行。クリアした場合のみ本番環境へデプロイ。

## 4. 「とどける」：Google Cloud 本番品質設計

数千のエージェントが同時に動く負荷に耐え、かつユーザーがリアルタイムに戦況を監視できるスケーラブルなインフラ設計。

### 使用コンポーネントと選定理由

| コンポーネント | 役割 | 選定理由（本番品質への必然性） |
| --- | --- | --- |
| **Cloud Run** | API / Agent Core | 完全サーバーレスで、シミュレーション開始時の一爆発的な並列アクセス（数千エージェントの同時起動）に対して高速にオートスケールする。 |
| **Cloud Pub/Sub** | タイムラインの配信バッファ | エージェント間のメッセージ伝播を非同期で媒介。システム全体の結合度を下げ、一部のエージェントの遅延が全体を止めない設計（疎結合）にする。 |
| **Firestore** | リアルタイム・データベース | シミュレーション中のタイムラインの更新を、クライアント（フロントエンド）へ **WebSocket/Server-Sent Events** を使ってリアルタイムに同期・描画する。 |
| **Vertex AI** | Gemini API | マルチモーダル対応かつ超高速。デマ画像（フェイク画像）に対するエージェントの反応をミリ秒単位で処理する。 |

## 5. ユーザーインターフェース（UI/UX）要件

ユーザー（企業の広報担当やリスク管理者）に「届ける」ための本番クオリティの画面設計。

* **ネットワーク・トポロジー・グラフ:** フェイクニュースがどのインフルエンサーエージェントを経由して、どのコミュニティ（クラスタ）に飛び火したかをリアルタイムに2D/3Dグラフで描画。
* **Datadog風「社会インフラ監視ダッシュボード」:**
* `拡散レート（RPS: Retweets Per Second）`
* `ユーザー感情比率（怒り・恐怖・冷静の割合）`
* `ブランド毀損度（0% - 100%）`


* **パッチ当てシミュレーション機能:** 画面上の「ファクトチェック文を投入」ボタンを押すと、カウンターエージェントが自律的に動き出し、拡散をどこまで食い止められるかをリアルタイムに追試（インジェクション）できるUI。
